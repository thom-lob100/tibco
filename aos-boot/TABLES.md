# 테이블 명세서 (aos-boot)

회사 DB에 aos-boot용 테이블을 생성할 때 DBA에 제출하는 명세서다.
원천 DDL은 `aos-boot-core/src/main/resources/schema.sql`(H2 문법, 개발용)이며,
운영에 필요한 테이블은 **`rv_command_queue` 1개**뿐이다 — 그것도
`@RvCommand(persistent = true)`(영속 커맨드 큐)를 쓰는 경우에만 필요하다.

패치 시스템(modern-patch)의 중앙 DB 테이블 7개는 별도 명세다 —
형제 저장소 `modern-patch`의 `docs/table-spec.md` 참조.

## 1. 타입 변환 규칙 (H2 → Oracle)

| H2 (schema.sql) | Oracle | 비고 |
|---|---|---|
| `BIGINT GENERATED BY DEFAULT AS IDENTITY` | `NUMBER(19) GENERATED BY DEFAULT AS IDENTITY` | Oracle 12c+; 그 이전 버전은 시퀀스+트리거 |
| `VARCHAR(n)` | `VARCHAR2(n CHAR)` | |
| `INT` | `NUMBER(10)` | |
| `CLOB` | `CLOB` | |
| `TIMESTAMP` | `TIMESTAMP` | |

- `IF NOT EXISTS`는 H2 편의 문법 — Oracle에서는 제거하고 생성한다.
- 운영 DB에서는 `spring.sql.init.mode`를 제거(또는 `never`)한다 (USAGE.md §2-3).

## 2. rv_command_queue — 영속 커맨드 재처리 큐

`persistent = true`로 선언된 command의 메시지를 **처리 전에** 저장해 두는 큐.
처리에 실패해도 (JVM이 죽어도) 행이 남아 있어 백그라운드 폴러가 재시도한다.
여러 인스턴스가 같은 테이블을 공유해도 안전하다 (조건부 UPDATE로 행을 선점).

| 컬럼 | 타입(H2) | NULL | 설명 |
|---|---|---|---|
| `id` | BIGINT IDENTITY | N | PK |
| `command` | VARCHAR(100) | N | subject 마지막 요소 (어느 핸들러로 갈지) |
| `subject` | VARCHAR(500) | N | 수신 당시 subject 원문 |
| `payload` | CLOB | N | 메시지 필드의 JSON 직렬화 |
| `status` | VARCHAR(20) | N | `PENDING`(대기) \| `PROCESSING`(처리 중) \| `FAILED`(dead letter). 처리 성공한 행은 **삭제**되므로 완료 상태값은 없다 |
| `attempts` | INT | N | 시도 횟수 |
| `max_attempts` | INT | N | 최대 시도 횟수 (초과 시 FAILED) |
| `next_attempt_at` | TIMESTAMP | N | 다음 재시도 예정 시각 (폴러 픽업 기준) |
| `last_error` | VARCHAR(2000) | Y | 마지막 실패 원인 |
| `created_at` | TIMESTAMP | N | 최초 저장 시각 |
| `updated_at` | TIMESTAMP | N | 마지막 상태 변경 시각 |

**필수 인덱스** (폴러가 주기적으로 조회하는 조건):

```sql
CREATE INDEX idx_rv_command_queue_poll ON rv_command_queue (status, next_attempt_at);
```

운영 참고 (USAGE.md §12에 상세):

```sql
SELECT status, COUNT(*) FROM rv_command_queue GROUP BY status;       -- 적체 확인
SELECT * FROM rv_command_queue WHERE status = 'FAILED';              -- dead letter 조회
UPDATE rv_command_queue SET status='PENDING', attempts=0 WHERE id=?; -- 수동 재처리
```

성공한 행은 즉시 삭제되므로 평상시 테이블은 작게 유지된다 — 별도 보관 정책이 필요한
것은 `FAILED`(dead letter) 행뿐이며, 원인 조치 후 재처리하거나 수동 삭제한다.

## 3. 샘플 전용 테이블 — 운영 생성 불필요

`sample_settlement`, `eqp_master`는 `aos-boot-samples`(데모 모듈) 전용이며
운영 산출물(`aos-boot-application-*-exec.jar`)에는 포함되지 않는다 — **회사 DB에 만들지 않는다.**
